# 얼마드나 — 단종업체 광고 시스템 개발 명세

> **목적**: 얼마드나의 광고 수익 모델 중 "단종업체(철거·전기·목공·도배·타일·샷시 등) 대상 광고 시스템"을 개발하기 위한 명세서.
> **대상**: Claude Code CLI (개발 세션)
> **상위 문서**: `얼마드나_프로젝트_마스터브리핑.md` (프로젝트 전반은 그쪽 참조)
> 작성: 도토리스튜디오 / 2026.05

## 0. 한 줄 요약

얼마드나는 **추천·매칭을 하지 않는다.** 광고 = 지면 판매다.
이 시스템은 단종업체가 (1) 지역×공정 단위로 광고 지면을 사고, (2) 무료 체험 후 유료 전환하고, (3) 다른 업체를 추천해 할인받는 — 셀프서비스 광고 플랫폼이다.

---

## 1. 사업 배경 (개발자가 알아야 할 맥락)

### 1.1 왜 매칭이 아니라 광고인가
- **매칭(집닥 방식)**: 소비자에게 특정 업체를 골라 연결 → 그 업체가 사고 치면 책임 사슬 발생 (CS·분쟁·평판 리스크).
- **광고(얼마드나 방식)**: 지면만 판매 → "이 배너 업체가 사고 쳐도 얼마드나는 책임지지 않는다" (푸터 면책 문구로 종결).
- **결론**: 얼마드나는 어떤 경우에도 업체를 "추천"하거나 "선별"하지 않는다. 견적 결과와 특정 업체를 연동해 노출하는 것도 금지(그 순간 법적·인식상 매칭이 됨). **순수 지면 판매만 한다.**

### 1.2 집닥 대비 구조적 우위
- 집닥은 **턴키 인테리어 업체만** 매칭한다. 소비자 1명 = 업체 1건으로 거래가 닫힌다.
- 얼마드나는 **턴키 + 모든 단종 직군 + 자재 브랜드**에게 동시에 지면을 판다.
- 같은 소비자 1명의 트래픽을 → 결과 페이지 배너(턴키), 공정별 슬롯(철거/전기/목공/샷시 등), PDF 내 지면(자재 브랜드)로 **다중 판매**.
- 광고주 모집단 = (턴키 + 전 단종 직군) × 51개 지역.

### 1.3 단종업체 영업 시 핵심 난관 — 개발에 영향을 줌
- 단종업체 사장님이 광고를 망설이는 진짜 이유는 가격이 아니라 **"0건이면 생돈 날림"이라는 공포**다.
- 매칭은 성과 연동(계약 따야 수수료)이라 이 공포가 없지만, 지면팔이는 결과 무관 선불이라 공포가 크다.
- 따라서 시스템은 이 공포를 완화하는 3개 장치를 **기능으로** 내장해야 한다:
  1. **무료 체험 기간** (첫 달 또는 첫 분기)
  2. **콜 기반 과금 옵션** — 노출 무료 + 전화 클릭/연결당 과금 (※ 이것은 매칭이 아님. 단순 광고 성과 측정이므로 책임 사슬 없음)
  3. **세그먼트 데이터 대시보드** — "당신이 산 슬롯(지역×공정)에 지난 30일 몇 명이 관심을 보였는가"를 숫자로 제시

## 2. 광고 상품 정의 (개발해야 할 데이터 모델)

### 2.1 광고주(Advertiser) 유형
| 유형 | 예시 | 객단가 | 특징 |
|---|---|---|---|
| 자재 브랜드 | KCC, LX, 동화 | 높음 | 적은 수, 안정적, 지면 광고 익숙 → 매출 기둥 |
| 턴키 인테리어 업체 | 지역 종합 인테리어 | 중간 | 결과 페이지 상단 배너 |
| 단종업체 | 철거/전기/목공/도배/타일/샷시/마루/도장/조명/청소 | 낮음 | 머릿수 많음(롱테일), 과금 저항 큼 → 가격·과금 설계가 관건 |

### 2.2 광고 상품 2종 (단종업체용)
| 상품 | 과금 방식 | 설명 |
|---|---|---|
| **고정형** | 월정액 (예: 월 2만 원대) | 지역×공정 슬롯에 배너 노출. 결과와 무관하게 선불 |
| **콜 기반형** | 노출 무료 + 전화 클릭/연결당 과금 (예: 클릭당 5천~1만 원) | "전화 와야 돈 낸다" → 0건 공포 제거. **추천 아님 = 책임 없음** |

> 가격 수치는 잠정값. 코드에서는 하드코딩하지 말고 **관리자가 변경 가능한 설정값(DB/설정 테이블)**으로 둘 것.

### 2.3 슬롯 구조
- 슬롯 = **공정 카테고리 × 지역**의 조합.
- 공정 카테고리: 철거, 설비, 전기, 목공, 도배, 타일, 샷시, 마루, 도장, 가구, 조명, 청소 등 (얼마드나 견적 엔진의 공정 분류와 일치시킬 것).
- 지역: 최대 51개 권역 (IP 기반 지역 판별 + 사용자가 서비스 내에서 선택한 지역값 우선).
- 동일 슬롯(예: "남양주 / 철거")에는 원칙적으로 광고주 1팀(또는 소수 로테이션).

## 3. 개발해야 할 핵심 기능

### 3.1 광고주 셀프서비스 관리 페이지
- 광고주가 직접: 배너 이미지 업로드, 클릭 URL 등록, 광고 지역 선택, 공정 슬롯 선택, 게재 기간 설정.
- 배너 소재 규격 검증 (사이즈/용량 — 디자인 가이드 참조).
- 등록 후 관리자 검수 → 승인 → 게재 (부적절·불법 광고 거부권).

### 3.2 광고 송출 로직 (지역 타겟)
- 사용자 IP로 접속 지역 판별 → 서비스 내 지역 선택값이 있으면 그 값 우선.
- 해당 지역 × 해당 공정 슬롯에 매칭되는 배너 송출.
- 빈 슬롯 폴백 처리 (초기엔 구글 애드센스 폴백 또는 자체 안내 배너).
- **금지**: 견적 결과(특정 업체 적합도 등)와 배너를 연동하지 말 것. 지역×공정 외의 기준으로 특정 업체를 노출하면 "추천"이 되어 책임 사슬 발생.

### 3.3 콜 트래킹 (콜 기반 과금용)
- 배너 내 전화 버튼 클릭 수 카운팅.
- 클릭 이벤트 → 광고주별 집계 → 과금 계산.
- ※ 이것은 광고 성과 측정이지 매칭이 아님. 업체를 골라주는 행위가 아니라 "노출된 배너가 몇 번 눌렸나"를 셀 뿐.

### 3.4 세그먼트 데이터 대시보드 (영업의 핵심 무기)
광고주에게 "당신이 산(또는 살) 슬롯의 트래픽"을 보여준다. **총량이 아니라 세그먼트 단위로** 보여줘야 한다.

수집·집계할 지표:
- 동시 접속자 수
- 월간 접속자 수 (MAU)
- 견적서 작성 횟수
- 견적서 공유 횟수
- 견적서 다운로드 횟수 (PDF/엑셀/이미지)

**표시 단위**: 위 지표를 전부 **지역 × 공정 카테고리 × 기간(최근 30일 등)**으로 쪼개서 보여줄 것.
- 예: "남양주 지역 / 철거 공정 클릭 / 최근 30일 / 84명"
- "월 5만 명 방문" 같은 총량 숫자는 단종업체 영업에 효과가 없다. 사장님은 "우리 동네, 우리 공정에 몇 명?"만 궁금하다.

**수요 강도 구분**: 단순 "견적 조회"와 "견적서 다운로드·공유"를 별도 지표로 분리할 것.
- 다운로드·공유한 사용자 = 실제 진행 단계의 강한 수요층.
- 영업 자료에서 이 층을 따로 강조하므로, 대시보드에서도 별도 카운트로 보여줘야 함.

### 3.5 무료 체험 관리
- 신규 광고주에게 첫 달(또는 첫 분기) 무료 기간 부여.
- 무료 기간 종료 시점 → 유료 전환 유도 플로우.
- 무료/유료 상태를 광고주 계정에 명확히 플래그.

### 3.6 추천 리워드 시스템 ★ (이번 갱신의 핵심 — 정확히 구현할 것)
**중요 — 잘못 만들면 무료 사용자만 양산하는 기계가 된다.**

- 추천 리워드는 **무료 기간 연장이 아니다.**
- 리워드 지급 조건: **피추천 업체가 "유료 결제"를 완료했을 때만** 추천한 업체에게 리워드 지급.
- 리워드 형태: 추천한 업체의 **다음 달 광고비 50% 할인** (또는 동등 가치 — 설정값으로).
- 구현 로직:
  1. 각 광고주에게 고유 추천 코드/링크 발급.
  2. 신규 광고주가 추천 코드를 통해 가입 → "추천인" 정보를 신규 계정에 기록.
  3. 신규 광고주가 **무료 체험이 아니라 실제 유료 결제**를 완료하는 이벤트 발생.
  4. 그 시점에 → 추천인 계정에 리워드(다음 결제 50% 할인) 적용.
- **무료 체험(3.5)과 추천 리워드(3.6)는 완전히 분리된 기능이다.** 무료 체험은 신규 광고주 자신을 위한 1번 카드, 추천 리워드는 유료 전환과 묶인 별개 보상. 코드·DB에서도 섞지 말 것.

### 3.7 PDF/엑셀/이미지 견적서 내 광고 지면
- 견적서 다운로드물(PDF/엑셀/PNG) 안에 광고 지면 삽입.
- 이 지면은 구글이 접근 불가한 독점 지면 → 별도 광고 상품으로 관리.
- 견적서가 공유·재열람될 때마다 광고가 노출되는 구조 (영구 노출 특성).

## 4. 결제 / 정산 (별도 인프라 — 마스터브리핑 §7 참조)

광고주 셀프서비스 결제를 위해 필요한 것:
- PG사: 정기결제(빌링) 지원 필수 — 토스페이먼츠 권장.
- 통신판매업 신고, 세금계산서 자동발행(포트원/팝빌/바로빌) 연동 — 광고주가 B2B이므로 필수.
- 결제 시스템은 (1) 고정형 월정액 정기결제, (2) 콜 기반형 사용량 과금, (3) 추천 리워드 할인 적용 — 세 가지를 모두 처리할 수 있어야 함.

## 5. 영업 메시지 (참고 — 기능 설계의 근거)

개발자가 "왜 이 기능이 필요한지" 이해하도록 영업 측 논리를 첨부한다. 단종업체에게 전달하는 5개 메시지:

1. **데이터로 증명** — 지역×공정 세그먼트 트래픽 숫자 제시 (→ 3.4 대시보드)
2. **리드의 질** — "견적서를 다운로드·공유한 사람 = 검증된 강한 수요층" (→ 3.4 수요 강도 구분)
3. **지역 타겟 노출** — 우리 동네 손님한테만 노출 (→ 3.2 송출 로직). ※ "매칭"이라 부르지 않음
4. **견적서 영구 노출** — 공유될 때마다 모르는 소비자에게까지 광고가 따라다님 (→ 3.7 PDF 지면)
5. **무료 체험 + 추천 리워드** — 못 믿겠으면 공짜 체험, 추천은 유료 결제와 묶음 (→ 3.5, 3.6)

## 6. 개발 시 절대 원칙 (요약)

1. **추천·선별 금지.** 얼마드나는 지면만 판다. 견적 결과와 업체를 연동하지 말 것.
2. 트래픽 지표는 **총량이 아니라 지역×공정 세그먼트**로 보여줄 것.
3. **단순 조회 vs 다운로드·공유**를 별도 지표로 분리할 것.
4. 콜 트래킹은 광고 성과 측정이지 매칭이 아님 (책임 사슬 없음).
5. **무료 체험과 추천 리워드는 별개 기능.** 추천 리워드는 **피추천자의 유료 결제** 시에만 지급.
6. 가격·요율은 하드코딩 금지 — 관리자 설정값으로.
